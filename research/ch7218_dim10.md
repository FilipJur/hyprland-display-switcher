# Research: AMDGPU Display Code Differences Between Kernel 6.18 and 7.0

## Executive Summary

The user's observed MCLK behavior difference (96MHz on 6.18 vs 456MHz on 7.0 in single-display mode) is **most likely explained by changes in how the AMDGPU driver handles idle optimizations and DPM power profile pausing between kernel versions**. The key commits involve:

1. **"drm/amd/display: pause the workload setting in dm"** (`50f29ead1f1b`) — introduced problematic `amdgpu_dpm_pause_power_profile()` calls around idle optimization
2. **Revert of the above** (`1412482b7143`) — fixed audio/performance/clock regressions in 6.19-rc
3. **"drm/amd/display: Add Idle state manager(ISM)"** (`0b572a6bf04a`) — refactored idle optimization logic into `amdgpu_dm_ism.c`
4. **"Undo accidental fix revert in amdgpu_dm_ism.c"** (April 2026) — fixed ISM which accidentally reintroduced the problematic pause_power_profile calls

CachyOS does **not** appear to carry specific amdgpu display patches in their `kernel-patches` repository beyond upstream code.

## 2. Critical Upstream Commits Affecting MCLK / Display Power Management

### 2.1 The "Pause Workload Setting" Commit (Problematic)

**Commit**: `50f29ead1f1b`  
**Subject**: `drm/amd/display: pause the workload setting in dm`  
**Author**: Kenneth Feng <kenneth.feng@amd.com>  
**Date**: October 2025  
**File**: `drivers/gpu/drm/amd/display/amdgpu_dm/amdgpu_dm_crtc.c`

**Code change**:
```c
static void amdgpu_dm_crtc_vblank_control_worker(struct work_struct *work)
{
    ...
    if (dm->active_vblank_irq_count == 0) {
        dc_post_update_surfaces_to_stream(dm->dc);

        r = amdgpu_dpm_pause_power_profile(adev, true);
        if (r)
            dev_warn(adev->dev, "failed to set default power profile mode\n");

        dc_allow_idle_optimizations(dm->dc, true);

        r = amdgpu_dpm_pause_power_profile(adev, false);
        if (r)
            dev_warn(adev->dev, "failed to restore the power profile mode\n");
    }
    ...
}
```

**What it does**: Pauses DPM (Dynamic Power Management) power profiles during idle optimization (`dc_allow_idle_optimizations`).

**Why it matters for MCLK**: The `amdgpu_dpm_pause_power_profile()` calls can interfere with the SMU's ability to manage memory clocks. When the power profile is paused, the SMU may not properly transition MCLK between power states, causing it to stick at either minimum (96MHz) or an intermediate state.

### 2.2 The Revert (Fix)

**Commit**: `1412482b7143`  
**Subject**: `Revert "drm/amd/display: pause the workload setting in dm"`  
**Merged**: For Linux 6.19-rc7 (January 2026)  
**Bug reports fixed**:
- HDMI Audio — missing start of streams / speakers produce "click" when idle
- 7900 XTX falls from bus with newer kernels
- Performance regression on 7900XTX
- GPU Clocks causing HDMI audio dropouts
- Stuttering Video when running other GUI-program in parallel RX7600

**Source**: Phoronix — "AMDGPU Driver Reverts Code For A Number Of Regressions On Linux 6.19" (2026-01-24)  
**URL**: https://www.phoronix.com/news/AMDGPU-Linux-6.19-Regressions

### 2.3 The Idle State Manager (ISM) Refactor

**Commit**: `0b572a6bf04a`  
**Subject**: `drm/amd/display: Add Idle state manager(ISM)`  
**File**: `drivers/gpu/drm/amd/display/amdgpu_dm/amdgpu_dm_ism.c`

**What it does**: Moves idle optimization logic from `amdgpu_dm_crtc.c` (`amdgpu_dm_crtc_vblank_control_worker`) to a new ISM module using a delayed worker + hysteresis to prevent idle state thrashing.

**Critical issue**: The ISM refactor **accidentally lost the fix** from commit `1412482b7143`. The problematic `amdgpu_dpm_pause_power_profile()` calls were reintroduced in the new `dm_ism_commit_idle_optimization_state()` function.

### 2.4 The ISM Fix (April 2026)

**Patch**: `drm/amd/display: Undo accidental fix revert in amdgpu_dm_ism.c`  
**Author**: Leo Li <sunpeng.li@amd.com>  
**Date**: April 17, 2026  
**Fixes**: `0b572a6bf04a` ("drm/amd/display: Add Idle state manager(ISM)")

**Code change** (removes the problematic pause_power_profile calls):
```c
// REMOVED from dm_ism_commit_idle_optimization_state():
- int r;
- r = amdgpu_dpm_pause_power_profile(adev, true);
- if (r)
-     dev_warn(adev->dev, "failed to set default power profile mode\n");
  dc_allow_idle_optimizations(dm->dc, true);
- r = amdgpu_dpm_pause_power_profile(adev, false);
```

**Mailing list discussion**:
> Alex Deucher: "Do you think ISM will fix this so we can re-enable pausing the power profile?"
> 
> Leo Li: "This regression was caught in our testing this week, so I don't think so. The ISM doesn't differ from the original vblank on/off logic much."

**Source**: amd-gfx mailing list, April 2026  
**URL**: https://www.mail-archive.com/amd-gfx@lists.freedesktop.org/msg141402.html

## 3. How This Explains the 96MHz vs 456MHz MCLK Difference

### 3.1 MCLK States on AMDGPU
From Arch Wiki AMDGPU documentation:
```
$ cat /sys/class/drm/card0/device/pp_dpm_mclk
0: 96Mhz *
1: 456Mhz
2: 675Mhz
3: 1000Mhz
```

- **96MHz** = deepest power-saving MCLK state (P-state 0)
- **456MHz** = intermediate MCLK state (P-state 1)

### 3.2 Behavior by Kernel Version

| Kernel | Idle Optimization | DPM Power Profile Pausing | Expected MCLK (single display, idle) |
|--------|-------------------|---------------------------|--------------------------------------|
| 6.18 LTS | `dc_allow_idle_optimizations()` enabled | **NOT present** (original commit not in 6.18, or reverted in stable) | Likely **96MHz** (proper idle downclocking) |
| 6.19 | `dc_allow_idle_optimizations()` enabled | **REVERTED** (`1412482b7143`) | Varies — depends on whether revert is present |
| 7.0 | ISM (`amdgpu_dm_ism.c`) | **Depends on whether ISM fix is included** | If ISM fix absent: may stick at **456MHz** or behave erratically |

### 3.3 Hypothesis for User's Observation

**Scenario A: CachyOS 6.18.24-lts**
- Based on 6.18 stable branch
- The "pause workload" commit `50f29ead1f1b` was **not backported** to 6.18 (it targeted 6.19/7.0 merge window)
- OR the revert `1412482b7143` **was backported** to 6.18 stable
- Idle optimizations work normally → MCLK drops to **96MHz** in single-display idle

**Scenario B: CachyOS 7.0.1**
- Based on 7.0 stable branch
- Contains ISM (`0b572a6bf04a`) which was merged during 7.0 development
- The "Undo accidental fix revert" patch may or may not be in 7.0.1
  - If **NOT in 7.0.1**: ISM still has `amdgpu_dpm_pause_power_profile()` calls
  - The pause_power_profile interferes with SMU MCLK decisions
  - Result: MCLK stays at **456MHz** instead of dropping to 96MHz
  - OR: the SMU, seeing the paused profile, keeps MCLK at a safe intermediate level

**Alternative explanation**: The ISM hysteresis/delayed worker changes the timing of when idle optimizations are applied. In single-display mode, the ISM may be more conservative about allowing deep idle states, keeping MCLK at 456MHz to ensure display stability.

**Subject**: `drm/amd/display: Bump the HDMI clock to 340MHz`  
**Author**: Mario Limonciello <mario.limonciello@amd.com>  
**Files**:
- `drivers/gpu/drm/amd/display/dc/dc_hdmi_types.h`

**Change**:
```c
-#define DP_ADAPTOR_HDMI_SAFE_MAX_TMDS_CLK 165000
+#define DP_ADAPTOR_HDMI_SAFE_MAX_TMDS_CLK 340000
```

**Impact**: For users with DP-to-HDMI dongles, this allows higher resolution/refresh rates. Not directly related to MCLK behavior but could affect link bandwidth calculations.

**Backported to 6.18 stable**: Yes (seen in 6.18.7 changelog)

### 4.2 PCON VRR ID Check Patch

**Patch**: `DC_OVERRIDE_PCON_VRR_ID_CHECK`  
**Status**: Exists in both CachyOS 6.19 and 7.0 patch sets (mentioned in search results)  
**Impact**: Allows Variable Refresh Rate (VRR/FreeSync) over HDMI 2.1 FRL via PCON (Protocol Converter). This could affect display link configuration and power management decisions.

## 7. Recommendations for User

### 7.1 Immediate Workarounds

**Option 1**: Force MCLK to desired level via sysfs
```bash
# Check available levels
cat /sys/class/drm/card0/device/pp_dpm_mclk

# Force manual control
echo "manual" > /sys/class/drm/card0/device/power_dpm_force_performance_level

# Lock to intermediate or high MCLK
echo "1" > /sys/class/drm/card0/device/pp_dpm_mclk  # 456MHz
echo "3" > /sys/class/drm/card0/device/pp_dpm_mclk  # 1000MHz (or max)
```

**Option 2**: Use `dcdebugmask` to disable idle optimizations
```bash
# Disable all Idle Power States
amdgpu.dcdebugmask=0x800  # DC_DISABLE_IPS

# Or disable IPS2 dynamic (keeps IPS1)
amdgpu.dcdebugmask=0x1000  # DC_DISABLE_IPS2_DYNAMIC

# Force enable all IPS (opposite effect)
amdgpu.dcdebugmask=0x2000  # DC_FORCE_IPS_ENABLE
```

**Option 3**: Check CachyOS 7.0.1 ISM patch status
```bash
# Check if the ISM fix is present in your kernel
grep -r "amdgpu_dpm_pause_power_profile" /sys/kernel/debug/dri/
# Or check kernel source if available
grep -n "amdgpu_dpm_pause_power_profile" drivers/gpu/drm/amd/display/amdgpu_dm/amdgpu_dm_ism.c
```

### 7.2 Long-Term Fix

If CachyOS 7.0.1 does not include the April 2026 "Undo accidental fix revert in amdgpu_dm_ism.c" patch, the user should:
1. Report to CachyOS maintainers
2. Request backport of the ISM fix
3. Or downgrade to 6.18 LTS until fixed
