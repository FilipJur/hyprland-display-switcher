# Research Dim 6: CH7218 EDID/DPCD Requirements for Stable FRL

## Executive Summary

The Chrontel CH7218A is a DP 1.4a to HDMI 2.1 Protocol Converter (PCON) with embedded MCU, ROM, and EDID buffer. It supports HDMI 2.1 FRL (Fixed Rate Link) mode up to 48 Gbps. This research investigates whether EDID/DPCD quirks or overrides can stabilize FRL operation on this adapter.

**Key Finding**: There are NO CH7218-specific EDID or DPCD quirks in the upstream Linux kernel. The only CH7218-specific kernel handling is the VRR PCON whitelist (branch device ID `0x2B02F0`). EDID override mechanisms exist at both kernel boot parameter and debugfs levels, but DPCD manipulation requires either amdgpu-specific debugfs interfaces or third-party tools.

## 1. CH7218-Specific Kernel Quirks

### 1.1 VRR PCON Whitelist (FOUND)

The **only** CH7218-specific handling in the amdgpu kernel driver is the VRR (Variable Refresh Rate) PCON whitelist.

**Patch**: `[PATCH] amdgpu: Add CH7218 PCON to the VRR whitelist`  
**Upstream Issue**: https://gitlab.freedesktop.org/drm/amd/-/issues/4773  
**Branch Device ID**: `0x2B02F0` (defined as `DP_BRANCH_DEVICE_ID_2B02F0`)

```c
// drivers/gpu/drm/amd/display/include/ddc_service_types.h
#define DP_BRANCH_DEVICE_ID_2B02F0 0x2B02F0 /* Chrontel CH7218 */

// drivers/gpu/drm/amd/display/amdgpu_dm/amdgpu_dm_helpers.c
static bool dm_is_freesync_pcon_whitelist(const uint32_t branch_dev_id)
{
    switch (branch_dev_id) {
    case DP_BRANCH_DEVICE_ID_0060AD:  // Synaptics VMM7100
    case DP_BRANCH_DEVICE_ID_00E04C:
    case DP_BRANCH_DEVICE_ID_90CC24:  // Synaptics Panamera
    case DP_BRANCH_DEVICE_ID_2B02F0:  // Chrontel CH7218
        ret_val = true;
        break;
    default:
        ret_val = false;
        break;
    }
    return ret_val;
}
```

**Note**: The CH7218 is explicitly whitelisted for FreeSync VRR passthrough. A follow-up patchset also enables generic HDMI VRR over PCON (issue #4805), which would allow HDMI-VRR on displays that don't advertise FreeSync.

### 1.2 DPCD Quirk Database (NOT FOUND)

The kernel's shared DPCD quirk database (`dpcd_quirk_list[]` in `drivers/gpu/drm/display/drm_dp_helper.c`) does **NOT** contain any entry for Chrontel or CH7218.

Current entries include:
- Analogix 7737 (`OUI 0x00,0x22,0xb9`): `DP_DPCD_QUIRK_LIMITED_M_N`
- Apple MacBookPro 2017 (`OUI 0x00,0x10,0xfa`): `DP_DPCD_QUIRK_CAN_DO_MAX_LINK_RATE_3_24_GBPS`
- Synaptics Panamera (`OUI 0x90,0xCC,0x24`): `DP_DPCD_QUIRK_DSC_THROUGHPUT_BPP_LIMIT`

**Status**: `NOT FOUND` - No CH7218-specific DPCD quirks exist in the kernel.

### 1.3 Vendor-Specific Handling in dc_link_dp.c

The 2021 FRL PCON support patch (`[PATCH v2] drm/amd/display: Add DP-HDMI FRL PCON Support in DC`) added generic PCON FRL handling to `dc_link_dp.c`:

```c
// drivers/gpu/drm/amd/display/dc/core/dc_link_dp.c
if (link->dc->caps.hdmi_frl_pcon_support) {
    link->dpcd_caps.dongle_caps.dp_hdmi_frl_max_link_bw_in_kbps =
        dc_link_bw_kbps_from_raw_frl_link_rate_data(
            hdmi_color_caps.bits.MAX_ENCODED_LINK_BW_SUPPORT);

    union hdmi_encoded_link_bw hdmi_encoded_link_bw;

    // Intersect reported max link bw support with the supported link rate post FRL link training
    if (core_link_read_dpcd(link, DP_PCON_HDMI_POST_FRL_STATUS,
            &hdmi_encoded_link_bw.raw, sizeof(hdmi_encoded_link_bw)) == DC_OK) {
        link->dpcd_caps.dongle_caps.dp_hdmi_frl_max_link_bw_in_kbps = 
            intersect_frl_link_bw_support(
                link->dpcd_caps.dongle_caps.dp_hdmi_frl_max_link_bw_in_kbps,
                hdmi_encoded_link_bw);
    }
}
```

This code is **generic** and applies to all PCON adapters, not CH7218-specific.

## 2. EDID Override Mechanisms

### 2.1 Kernel Boot Parameter: `drm.edid_firmware`

**Parameter**: `drm.edid_firmware=[<connector>:]<edid_path>`

**Usage Examples**:
```bash
# Force EDID for a specific connector
drm.edid_firmware=DP-1:edid/custom_edid.bin

# Multiple connectors
drm.edid_firmware=DP-1:edid/custom1.bin,DP-2:edid/custom2.bin
```

**Requirements**:
- File must be in `/lib/firmware/edid/` or `/usr/lib/firmware/edid/`
- Kernel must have `CONFIG_DRM_LOAD_EDID_FIRMWARE=y`
- For early KMS (initramfs), the EDID file must be included in the initramfs

**Built-in EDID Options** (kernel source `drivers/gpu/drm/drm_edid_load.c`):
| Resolution | Path |
|------------|------|
| 800x600 | `edid/800x600.bin` |
| 1024x768 | `edid/1024x768.bin` |
| 1280x1024 | `edid/1280x1024.bin` |
| 1600x1200 | `edid/1600x1200.bin` |
| 1680x1050 | `edid/1680x1050.bin` |
| 1920x1080 | `edid/1920x1080.bin` |

**Note**: Recent kernels (6.x) have removed some built-in EDIDs due to validity concerns for digital connectors.

### 2.2 Runtime debugfs EDID Override

**Path**: `/sys/kernel/debug/dri/<N>/<connector>/edid_override`

**Usage**:
```bash
# Write custom EDID to override at runtime
sudo cat /usr/lib/firmware/edid/custom_edid.bin > /sys/kernel/debug/dri/0/DP-1/edid_override

# Read current EDID
sudo cat /sys/class/drm/card0-DP-1/edid > current_edid.bin
```

**Connectors affected**: `DP-1`, `DP-2`, `HDMI-A-1`, `HDMI-A-2`, etc.

**Note**: For amdgpu, this requires the connector to be in a state where EDID can be overridden. Some users report that `edid_override` has no effect unless the display is re-detected (unplug/replug).

### 2.3 EDID Manipulation Tools

| Tool | Purpose | Platform |
|------|---------|----------|
| `edid-decode` | Decode binary EDID to human-readable format | Linux |
| `edid-generator` | Generate custom EDID from text spec | Linux |
| `get-edid` / `parse-edid` | Read and parse monitor EDID | Linux |
| `AW EDID Editor` | GUI editor for EDID files | Windows/Mac |
| `EdidCraft` | Online EDID analysis and generation | Web |
| `EDIDMaker` | C++ EDID generator | Cross-platform |
| `thyge/edid-editor` | Open-source EDID editor | GitHub |

**Custom EDID Creation**:
```bash
# Extract current EDID
sudo cat /sys/class/drm/card0-DP-1/edid > my_edid.bin

# Decode it
edid-decode my_edid.bin

# Modify with custom timings (edit the .S file)
cp tools/edid/1920x1080.S custom.S
# Edit custom.S with desired timings
make -C tools/edid custom.bin

# Install and use
sudo cp custom.bin /lib/firmware/edid/
# Add to kernel cmdline: drm.edid_firmware=DP-1:edid/custom.bin
```

## 3. DPCD Manipulation

### 3.1 amdgpu debugfs DPCD Interface (CRITICAL FINDING)

The amdgpu driver exposes a **debugfs interface for arbitrary DPCD read/write** on DP connectors:

**Paths** (per connector in `/sys/kernel/debug/dri/<N>/<connector>/`):
- `aux_dpcd_address` - Set target DPCD address (write 4-byte value)
- `aux_dpcd_size` - Set transaction size in bytes (write 4-byte value, max 256)
- `aux_dpcd_data` - Read/write DPCD data at the configured address

**Usage**:
```bash
# Set DPCD address to 0x3036 (DP_PCON_HDMI_POST_FRL_STATUS)
printf '\x36\x30\x00\x00' > /sys/kernel/debug/dri/0/DP-1/aux_dpcd_address

# Set read size to 1 byte
printf '\x01\x00\x00\x00' > /sys/kernel/debug/dri/0/DP-1/aux_dpcd_size

# Read the DPCD register
cat /sys/kernel/debug/dri/0/DP-1/aux_dpcd_data

# Write to DPCD (e.g., configure PCON)
printf '\x3f' > /sys/kernel/debug/dri/0/DP-1/aux_dpcd_data
```

**Important**: No concurrency protection. Rapid successive accesses can cause strange behavior.

**Patch Reference**: `[PATCH 26/26] drm/amd/display: Add debugfs dpcd interface` (David Francis, 2019)

### 3.2 Tegra debugfs DPCD Interface (Reference)

NVIDIA Tegra provides similar debugfs access:
```bash
cd /sys/kernel/debug/tegra_dp0/dpaux_dpcd
echo 0x300 > addr
echo 4 > num_bytes
cat data
```

### 3.3 Generic DRM DP AUX Chardev

**Kernel Config**: `CONFIG_DRM_DISPLAY_DP_AUX_CHARDEV`

When enabled, creates `/dev/drm_dp_auxN` devices that allow userspace to read/write arbitrary DPCD registers via the DP AUX channel.

**Note**: This is a generic kernel interface but may not be enabled by default on all distributions.

### 3.4 Userspace Tools for DPCD Access

| Tool | Description | Notes |
|------|-------------|-------|
| `ddcutil` | DDC/CI utility over I2C | Can read EDID but not DPCD directly |
| NXP DPCD Tool | Windows utility for NXP DP-VGA dongles | Vendor-specific, not for CH7218 |
| Unigraf DP CTS Tool | Professional DisplayPort test equipment | Commercial, expensive |
| Custom scripts | Using `/dev/drm_dp_auxN` or amdgpu debugfs | Requires root |

## 5. CH7218 EDID Buffer Behavior

### 5.1 EDID Buffer Architecture

According to the CH7218A datasheet:
- **Embedded Microcontroller, ROM and EDID Buffer**: The chip has internal EDID storage
- **Firmware update through Chrontel proprietary technologies**: EDID can be updated via vendor tools
- **Auto-boot and EDID buffer**: Firmware auto-loaded from embedded ROM handles DP input detection and HDMI connection detection

### 5.2 EDID Passthrough vs. Generated EDID

The CH7218 can operate in two modes:
1. **EDID Passthrough**: The PCON forwards the HDMI sink's EDID to the DP source
2. **Generated/Buffered EDID**: The PCON presents its own EDID to the DP source

The register `DP_PROTOCOL_CONVERTER_CONTROL_1` (0x3051) bit 1 (`DP_HDMI_EDID_PROCESSING_DISABLE`) may control this behavior, but this is **inferred** from the DPCD register definitions, not confirmed for CH7218 specifically.

### 5.3 Known EDID-Related Issues with CH7218

**Issue #1**: VRR only works with FreeSync displays (not HDMI-VRR displays)
- **Root Cause**: amdgpu only parses FreeSync capability from EDID, not HDMI-VRR
- **Fix**: Patchset available to parse HDMI VRR info from EDID (gitlab issue #4805)
- **Workaround**: Use a display that advertises FreeSync, or apply the HDMI-VRR patchset

**Issue #2**: AVR/Soundbar EDID stripping
- When connecting through an AVR or soundbar, FreeSync capabilities may be removed from the EDID
- This is a general HDMI ecosystem issue, not CH7218-specific

**Issue #3**: No specific CH7218 EDID corruption bugs found
- Unlike some other PCON adapters, the CH7218 does not have known EDID corruption issues
- It has been reported as "more stable" than Synaptics VMM7100-based adapters

## 6. Can Forcing a Specific EDID Stabilize FRL?

### 6.1 Theoretical Basis

FRL stability depends on:
1. **Link training success**: Proper DPCD negotiation between DP source and PCON
2. **Bandwidth validation**: The mode must fit within `dp_hdmi_frl_max_link_bw_in_kbps`
3. **Timing compliance**: The PCON must support the exact pixel clock and timings

**EDID factors that could affect FRL**:
- **Pixel clock / dot clock**: Must not exceed PCON FRL bandwidth
- **Blanking intervals**: Standard CTA-861 timings are usually safe
- **DSC requirements**: If DSC is needed, the EDID must advertise DSC capability
- **Color depth/format**: RGB vs YCbCr, 8bpc vs 10bpc/12bpc

### 6.2 EDID Modifications for FRL Stability

If FRL instability is observed, these EDID modifications could theoretically help:

1. **Reduce advertised maximum TMDS clock**: Force the PCON to use FRL instead of TMDS fallback
2. **Add CTA-861 block with HDMI 2.1 VRR info**: May help VRR negotiation
3. **Ensure DSC capability is advertised**: If DSC passthrough is needed
4. **Match exact timing parameters**: Htotal, Vtotal, pixel clock exactly matching a CTA-861 standard

### 6.3 Practical Recommendation

**NOT VERIFIED**: There is no documented evidence that EDID override specifically fixes CH7218 FRL instability. The CH7218 has been reported as "stable FRL out-of-the-box" in community testing. If FRL instability occurs:

1. **First check DPCD**: Read `0x3036` (POST_FRL_STATUS) and `0x303B` (TX_LINK_STATUS) to diagnose
2. **Check bandwidth validation**: Ensure mode fits within reported `dp_hdmi_frl_max_link_bw_in_kbps`
3. **Try EDID override as last resort**: Use a known-good EDID from a similar display

## 9. Recommendations for FRL Stability Investigation

1. **Read DPCD 0x3036 first**: Check `DP_PCON_HDMI_POST_FRL_STATUS` to verify FRL mode and trained bandwidth
2. **Read DPCD 0x303B**: Check `DP_PCON_HDMI_TX_LINK_STATUS` for link active and FRL ready
3. **Verify branch ID**: Read DPCD 0x503 to confirm `0x2B02F0` detection
4. **Check bandwidth math**: Ensure requested mode fits within `dp_hdmi_frl_max_link_bw_in_kbps`
5. **EDID override if needed**: Use `drm.edid_firmware` or debugfs `edid_override` to test with a known-good EDID
6. **Consider HDMI-VRR patch**: If display supports HDMI-VRR but not FreeSync
7. **Contact Chrontel**: For firmware updates if EDID buffer behavior is suspected
