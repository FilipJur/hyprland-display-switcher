# Writing to power_dpm_force_performance_level Without Root — Research Findings

## Executive Summary

Writing to `/sys/class/drm/card*/device/power_dpm_force_performance_level` **requires root privileges** by kernel design. There is **no native polkit/dbus interface** that exposes direct user-level write access to this sysfs file. However, several **workarounds exist** using udev rules (with caveats), systemd services running as root, and user-space daemon tools (LACT, CoreCtrl) that provide non-root GUI/CLI access via a root-privileged helper daemon.

---

## 1. polkit Rules for AMDGPU

### 1.1 Native AMDGPU polkit Actions: NOT FOUND

**There are NO native polkit `.rules` or `.policy` files for the amdgpu driver's sysfs interfaces.**

The amdgpu kernel driver exposes power management directly via sysfs. There is no PolicyKit action definition (e.g., `org.freedesktop.amdgpu.power-management`) registered by the kernel or any standard systemd service for GPU power controls.

**Verification**: Searching for amdgpu in `/usr/share/polkit-1/actions/` and `/etc/polkit-1/rules.d/` on standard distributions yields no results for GPU power management.

### 1.2 CoreCtrl polkit Rule (Reference Pattern)

While not for amdgpu directly, **CoreCtrl** (a GPU control GUI) uses a helper daemon that runs with root privileges via dbus. It provides a polkit rule template:

**File**: `/etc/polkit-1/rules.d/90-corectrl.rules`

```javascript
polkit.addRule(function(action, subject) {
    if ((action.id == "org.corectrl.helper.init" ||
         action.id == "org.corectrl.helperkiller.init") &&
        subject.local == true &&
        subject.active == true &&
        subject.isInGroup("your-user-group")) {
            return polkit.Result.YES;
    }
});
```

**For older polkit (< 0.106)**, use `.pkla` format:

**File**: `/etc/polkit-1/localauthority/50-local.d/90-corectrl.pkla`

```
[User permissions]
Identity=unix-group:*
Action=org.corectrl.*
ResultActive=yes
```

**Key Insight**: This is the **indirect approach** — the polkit rule grants the user permission to launch a **root-privileged helper daemon**, which then writes to sysfs on the user's behalf.

## 2. udev Rules for Writable sysfs

### 2.1 Standard ATTR-based udev Rule (KNOWN TO FAIL)

The commonly documented udev rule for setting power_dpm_force_performance_level at boot:

```
KERNEL=="card0", SUBSYSTEM=="drm", DRIVERS=="amdgpu", ATTR{device/power_dpm_force_performance_level}="high"
```

Or using the parent pci device:

```
SUBSYSTEM=="pci", DRIVER=="amdgpu", ATTR{power_dpm_force_performance_level}="low"
```

**STATUS: BROKEN on modern kernels/systemd**

Arch Linux bug #72655 documents that this rule produces:
```
card0: /etc/udev/rules.d/30-amdgpu-pm.rules:1 Failed to write ATTR{/sys/devices/pci0000:00/0000:00:02.1/0000:01:00.0/drm/card0/device/power_dpm_force_performance_level}, ignoring: Permission denied
```

The rule works when manually triggered with `sudo udevadm trigger` but fails during boot. This is because sysfs attributes for GPU power management are protected by the kernel and udev runs in a restricted context during early boot.

### 2.2 RUN+ chmod/chown Workaround (PARTIALLY WORKS)

A udev rule using `RUN+=` to execute chmod/chown on the sysfs file:

```
ACTION=="add", SUBSYSTEM=="drm", DRIVERS=="amdgpu", \
    RUN+="/bin/sh -c '/bin/chmod 0666 /sys/class/drm/%k/device/power_dpm_force_performance_level; /bin/chown root:video /sys/class/drm/%k/device/power_dpm_force_performance_level'"
```

Or matching the PCI parent:
```
SUBSYSTEM=="pci", DRIVER=="amdgpu", ACTION=="add", \
    RUN+="/bin/sh -c '/bin/chmod 0666 /sys%p/power_dpm_force_performance_level; /bin/chown root:video /sys%p/power_dpm_force_performance_level'"
```

**Important Caveats**:
- The `chmod`/`chown` on sysfs files may be restricted by kernel sysfs security policies
- Sysfs attribute permissions are controlled by the driver at creation time; `chmod` may not persist or may be ignored for sysfs files created with restricted mode bits
- On some kernels, sysfs files in `/sys/class/drm/` are created with `0644` or `0600` and the kernel prevents world-writable sysfs attributes (`S_IWUGO` is blocked at the sysfs layer)

### 2.3 ROCm-Style udev Rule (Device Nodes Only)

ROCm documentation provides udev rules for **device node access** (`/dev/kfd`, `/dev/dri/renderD*`), NOT for sysfs:

**File**: `/etc/udev/rules.d/70-amdgpu.rules`

```
KERNEL=="kfd", GROUP="video", MODE="0660"
SUBSYSTEM=="drm", KERNEL=="renderD*", GROUP="video", MODE="0660"
```

This grants GPU **compute/render access** to the `video` group but does NOT affect sysfs power management file permissions.

### 2.4 Recommended Custom udev Rule (Best Effort)

If attempting udev-based sysfs permission changes:

**File**: `/etc/udev/rules.d/99-amdgpu-pm.rules`

```
# Match the DRM card device when amdgpu driver is loaded
ACTION=="add", SUBSYSTEM=="drm", KERNEL=="card[0-9]*", DRIVERS=="amdgpu", \
    RUN+="/bin/sh -c '/bin/chmod 0664 /sys/%p/device/power_dpm_force_performance_level && /bin/chgrp video /sys/%p/device/power_dpm_force_performance_level'"
```

Then add your user to the `video` group:
```bash
sudo usermod -a -G video $USER
```

**Note**: This rule modifies the sysfs file permissions after the device appears. Effectiveness varies by kernel version and security configuration.

## 3. dbus Interfaces

### 3.1 UPower

**UPower does NOT expose GPU power controls.** UPower (`org.freedesktop.UPower`) provides battery, AC adapter, and general power management information but has no interface for GPU clock/power control.

### 3.2 systemd-logind

**systemd-logind** (`org.freedesktop.login1`) handles session/power-button management but does NOT expose GPU DPM controls.

### 3.3 LACT Daemon (Custom dbus/Unix Socket)

**LACT** (Linux AMDGPU Controller Tool) runs a daemon `lactd` as root. It exposes a Unix socket at a configurable location. The GUI connects to this socket. Permissions are controlled via the config file:

**File**: `/etc/lact/config.yaml`

```yaml
daemon:
  admin_group: wheel   # or video, sudo, etc.
  admin_user: foo      # optional specific user
```

This is **not a dbus service** but a custom Unix socket daemon that provides non-root access to root-privileged GPU control.

### 3.4 CoreCtrl Helper (dbus-activated)

CoreCtrl uses a **dbus-activated helper daemon** (`corectrl-helper`) that runs as root. The GUI communicates with this helper via dbus to execute sysfs writes.

## 4. User-Space Tools

### 4.1 rocm-smi / amd-smi — REQUIRES ROOT FOR WRITES

**Status**: `rocm-smi` and `amd-smi` require **sudo/root** for ALL write operations.

From ROCm documentation:
- `amd-smi reset` — "Requires 'sudo' privileges"
- `amd-smi set-perflevel` — requires sudo

From rocm_smi_lib issue #117:
```
ERROR: GPU[0]:RSMI_STATUS_PERMISSION: The user ID of the calling process does not have sufficient permission to execute a command.
```

Even `rocm-smi --show*` (read-only display) historically tried to relaunch with sudo (ROCm issue #562), though newer versions may allow reads without root.

**No setuid wrapper is provided** by ROCm for unprivileged writes.

### 4.2 LACT (Linux AMDGPU Controller Tool) — NON-ROOT GUI/CLI

**Status**: Provides **non-root access** via root daemon.

**Installation**:
```bash
# Arch
sudo pacman -S lact
# Fedora
sudo dnf install lact
# Debian/Ubuntu
sudo dpkg -i lact*.deb
```

**Setup**:
```bash
sudo systemctl enable --now lactd
```

**Key Features**:
- Performance levels (auto/low/high/manual)
- Power cap control
- Fan curves
- Overclocking (with `amdgpu.ppfeaturemask=0xffffffff`)
- Profile switching based on running processes
- Non-root GUI/CLI access (communicates with root daemon)

**Socket Permissions**:
By default uses `wheel` or `sudo` group. Configure in `/etc/lact/config.yaml`:
```yaml
daemon:
  admin_group: video
```

### 4.3 CoreCtrl — NON-ROOT GUI (with polkit setup)

**Status**: Provides **non-root access** via root helper daemon + polkit.

**Setup**:
1. Install CoreCtrl
2. Create polkit rule (see section 1.2)
3. Add `amdgpu.ppfeaturemask=0xffffffff` kernel parameter for overclocking
4. Copy `.desktop` to autostart

### 4.4 amdmond / amdgpud — MONITORING ONLY

The `amdmond` daemon (from the `amdgpud` project) is a **monitoring daemon** — it reads GPU metrics but does not provide control functionality without root.

### 4.5 amdgpu_top — READ-ONLY

`amdgpu_top` provides detailed real-time statistics but is **read-only**.

### 4.6 TuxClocker

`TuxClocker` is another Qt-based overclocking tool for AMD GPUs. Like CoreCtrl and LACT, it likely requires root privileges or a helper daemon for actual sysfs writes.

## 5. Kernel Parameters and ppfeaturemask

### 5.1 amdgpu.ppfeaturemask

**Parameter**: `amdgpu.ppfeaturemask` (hexint)

**Purpose**: Overrides which power play features are enabled. Default is the current set of stable power features.

**Location in source**: `drivers/gpu/drm/amd/include/amd_shared.h`

**Enum PP_FEATURE_MASK values**:

| Bit Mask | Name | Description |
|----------|------|-------------|
| `0x0001` | `PP_SCLK_DPM_MASK` | Dynamic adjustment of system (graphics) clock |
| `0x0002` | `PP_MCLK_DPM_MASK` | Dynamic adjustment of memory clock |
| `0x0004` | `PP_PCIE_DPM_MASK` | Dynamic adjustment of PCIE clocks and lanes |
| `0x0008` | `PP_SCLK_DEEP_SLEEP_MASK` | System clock deep sleep |
| `0x0010` | `PP_POWER_CONTAINMENT_MASK` | Power containment |
| `0x0020` | `PP_UVD_HANDSHAKE_MASK` | Unified video decoder handshake |
| `0x0040` | `PP_SMC_VOLTAGE_CONTROL_MASK` | Dynamic voltage control |
| `0x0080` | `PP_VBI_TIME_SUPPORT_MASK` | Vertical blank interval support |
| `0x0100` | `PP_ULV_MASK` | Ultra low voltage |
| `0x0200` | `PP_ENABLE_GFX_CG_THRU_SMU` | SMU control of GFX engine clockgating |
| `0x0400` | `PP_CLOCK_STRETCH_MASK` | Clock stretching |
| `0x0800` | `PP_OD_FUZZY_FAN_CONTROL_MASK` | Overdrive fuzzy fan control |
| `0x1000` | `PP_SOCCLK_DPM_MASK` | Dynamic adjustment of SoC clock |
| `0x2000` | `PP_DCEFCLK_DPM_MASK` | Dynamic adjustment of Display Controller Engine Fabric clock |
| `0x4000` | `PP_OVERDRIVE_MASK` | **Over- and under-clocking support** |
| `0x8000` | `PP_GFXOFF_MASK` | Dynamic graphics engine power control |
| `0x10000` | `PP_ACG_MASK` | Adaptive clock generator |
| `0x20000` | `PP_STUTTER_MODE` | Stutter mode |
| `0x40000` | `PP_AVFS_MASK` | Adaptive voltage and frequency scaling |
| `0x80000` | `PP_GFX_DCS_MASK` | GFX Async DCS |

### 5.2 Enabling OverDrive (Required for Clock/Voltage Control)

**Bit 14 (PP_OVERDRIVE_MASK = 0x4000)** must be set.

**Recommended safe value** (per Arch Wiki):
```bash
printf 'amdgpu.ppfeaturemask=0x%x\n' "$(($(cat /sys/module/amdgpu/parameters/ppfeaturemask) | 0x4000))"
```

**Full unlock** (enables all features, potentially unstable):
```
amdgpu.ppfeaturemask=0xffffffff
```

**Boot parameter methods**:

**GRUB** (`/etc/default/grub`):
```
GRUB_CMDLINE_LINUX_DEFAULT="... amdgpu.ppfeaturemask=0xffffffff"
```
Then: `sudo grub-mkconfig -o /boot/grub/grub.cfg`

**modprobe** (`/etc/modprobe.d/80-amdgpu.conf`):
```
options amdgpu ppfeaturemask=0xffffffff
```

### 5.3 Other Relevant Module Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `amdgpu.dpm` | int | Override DPM (0=disable, 1=enable, default=-1 auto) |
| `amdgpu.runpm` | int | Runtime PM for dGPUs (-1=auto, 0=disable, -2=auto with power down when displays attached) |
| `amdgpu.lockup_timeout` | int | GPU scheduler timeout in ms (default 2000) |
| `amdgpu.gpu_recovery` | int | GPU recovery mechanism |

## 6. sysfs Paths and Permissions Model

### 6.1 Exact sysfs Paths

```
/sys/class/drm/card0/device/power_dpm_force_performance_level
/sys/class/drm/card0/device/power_dpm_state          # legacy
/sys/class/drm/card0/device/pp_dpm_sclk
/sys/class/drm/card0/device/pp_dpm_mclk
/sys/class/drm/card0/device/pp_dpm_socclk
/sys/class/drm/card0/device/pp_dpm_fclk
/sys/class/drm/card0/device/pp_dpm_dcefclk
/sys/class/drm/card0/device/pp_dpm_pcie
/sys/class/drm/card0/device/pp_power_profile_mode
/sys/class/drm/card0/device/pp_od_clk_voltage
/sys/class/drm/card0/device/pp_table
/sys/class/drm/card0/device/hwmon/hwmon*/temp1_input
/sys/class/drm/card0/device/hwmon/hwmon*/power1_cap
/sys/class/drm/card0/device/gpu_busy_percent
/sys/class/drm/card0/device/mem_busy_percent
/sys/class/drm/card0/device/gpu_metrics
```

### 6.2 Kernel sysfs Permission Model

From the kernel source (`drivers/gpu/drm/amd/pm/amdgpu_pm.c`):

Sysfs PM attributes are created with mode bits that include `S_IWUGO` (write for user/group/others) for certain attributes, but the kernel's sysfs layer enforces restrictions:

```c
// From commit 11c9cc95f818f0f187e9b579a7f136f532b42445
// amdgpu/pm: Make sysfs pm attributes as read-only for VFs
dev_attr->attr.mode &= ~S_IWUGO;
dev_attr->store = NULL;
```

**Key fact**: The Linux kernel **does not allow world-writable (`666`) sysfs attributes** in general. The sysfs filesystem code enforces this security policy. Only the owner (root) can write unless the driver explicitly sets `S_IWGRP` or `S_IWOTH` at creation time.

For amdgpu power management sysfs files:
- Typically created as `0644` (owner read/write, group read, other read) or `0600`
- The owner is `root:root`
- Only root can write by default

### 6.3 sysfs Security Enforcement

From kernel sysfs documentation and the amdgpu driver behavior:
- sysfs file permissions are set at device creation time by the driver
- `chmod` on sysfs files may appear to work but the kernel may reject writes based on the original mode bits
- The kernel's sysfs internal permission check occurs before VFS-level permission checks

## 8. Recommended Solutions for Non-Root Access

### Solution A: LACT Daemon (Recommended)

1. Install LACT:
   ```bash
   # Arch: sudo pacman -S lact
   # Fedora: sudo dnf install lact
   ```

2. Enable daemon:
   ```bash
   sudo systemctl enable --now lactd
   ```

3. Configure group access:
   ```bash
   sudo usermod -a -G video $USER
   ```

4. Edit `/etc/lact/config.yaml`:
   ```yaml
   daemon:
     admin_group: video
   ```

5. Restart daemon and use GUI/CLI as non-root user

### Solution B: Custom systemd Service (Script-based)

Create a root-privileged service that applies settings at boot:

**File**: `/etc/systemd/system/amdgpu-pm.service`

```ini
[Unit]
Description=AMDGPU Power Management
After=systemd-modules-load.service

[Service]
Type=oneshot
ExecStart=/usr/local/bin/amdgpu-set-pm.sh
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target
```

**File**: `/usr/local/bin/amdgpu-set-pm.sh`

```bash
#!/bin/bash
until [ -f /sys/class/drm/card0/device/power_dpm_force_performance_level ]; do
    sleep 1
done
echo "high" > /sys/class/drm/card0/device/power_dpm_force_performance_level
echo "5" > /sys/class/drm/card0/device/pp_power_profile_mode
```

```bash
sudo chmod 755 /usr/local/bin/amdgpu-set-pm.sh
sudo systemctl enable --now amdgpu-pm.service
```

### Solution C: pkexec Wrapper Script

Create a script that uses `pkexec` (polkit's `org.freedesktop.policykit.exec`) to elevate:

**File**: `/usr/local/bin/set-gpu-perf`

```bash
#!/bin/bash
pkexec bash -c "echo '$1' > /sys/class/drm/card0/device/power_dpm_force_performance_level"
```

**File**: `/usr/share/polkit-1/actions/org.local.set-gpu-perf.policy`

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE policyconfig PUBLIC "-//freedesktop//DTD PolicyKit Policy Configuration 1.0//EN"
 "http://www.freedesktop.org/standards/PolicyKit/1/policyconfig.dtd">
<policyconfig>
  <action id="org.local.set-gpu-perf">
    <description>Set GPU performance level</description>
    <message>Authentication is required to set GPU performance level</message>
    <defaults>
      <allow_any>no</allow_any>
      <allow_inactive>no</allow_inactive>
      <allow_active>yes</allow_active>
    </defaults>
    <annotate key="org.freedesktop.policykit.exec.path">/usr/local/bin/set-gpu-perf</annotate>
  </action>
</policyconfig>
```

**Note**: This still requires authentication but provides a structured polkit-mediated path.

### Solution D: udev RUN+ Rule (Experimental)

```
ACTION=="add", SUBSYSTEM=="drm", KERNEL=="card[0-9]*", DRIVERS=="amdgpu", \
    RUN+="/bin/sh -c '/bin/chmod g+w /sys/%p/device/power_dpm_force_performance_level && /bin/chgrp video /sys/%p/device/power_dpm_force_performance_level'"
```

**Limitation**: May not work on kernels with strict sysfs security. The kernel may ignore the chmod on sysfs attributes.
