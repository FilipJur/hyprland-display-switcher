Research Document: PCON FRL Link Training & `read_and_intersect_post_frl_lt_status`
==========================================================================================

## 1. Exact Function Location and Signature

**Function**: `read_and_intersect_post_frl_lt_status`
**Signature**: `static void read_and_intersect_post_frl_lt_status(struct dc_link *link)`
**File**: `drivers/gpu/drm/amd/display/dc/link/protocols/link_dp_capability.c`
**Line**: 1116 (as seen in kernel ~6.8+ source via codebrowser.dev)

**Caller chain**:
1. `retrieve_link_cap(struct dc_link *link)` in `link_dp_capability.c` (or `dc_link_dp.c` in older kernels)
2. → calls `get_active_converter_info(uint8_t data, struct dc_link *link)` at line 1158
3. → `get_active_converter_info()` calls `read_and_intersect_post_frl_lt_status(link)` at line 1251

**Historical evolution**:
- **2021-12-01**: Original PCON FRL support added in `dc_link_dp.c` by Fangzhi Zuo. The function did not exist as a separate function; the PCON FRL logic was inline inside `get_active_converter_info()`.
- **2023-01-17**: Wenjing Liu moved DP link training logic to `link_dp_training` (refactoring of DC link code).
- **~2023-2024**: The PCON FRL logic was extracted into a standalone static helper function `read_and_intersect_post_frl_lt_status()` in the new `link_dp_capability.c` file.

## 2. DPCD Registers Read by the Function

The function reads three DPCD registers related to PCON FRL link training:

### Register 1: DP_REGULATED_AUTONOMOUS_MODE_SUPPORTED_AND_HDMI_LINK_TRAINING_STATUS
- **DPCD Address**: `0x303C`
- **Purpose**: Determines if the PCON dongle supports "regulated autonomous mode"
- **Bit of interest**: `REGULATED_AUTONOMOUS_MODE_SUPPORTED` (bit 7)
- **Definition**:
```c
union autonomous_mode_and_frl_link_status {
    struct {
        uint8_t FRL_LT_IN_PROGRESS_STATUS:1;
        uint8_t FRL_LT_LINK_CONFIG_IN_PROGRESS:3;
        uint8_t RESERVED:1;
        uint8_t FALLBACK_POLICY:1;
        uint8_t FALLBACK_POLICY_VALID:1;
        uint8_t REGULATED_AUTONOMOUS_MODE_SUPPORTED:1;  // Bit 7
    } bits;
    uint8_t raw;
};
```

### Register 2: DP_PCON_HDMI_TX_LINK_STATUS
- **DPCD Address**: `0x303B`
- **Purpose**: Reports the HDMI TX link status from the PCON
- **Bit of interest**: `HDMI_TX_READY_STATUS` (bit 1)
- **Definition**:
```c
union hdmi_tx_link_status {
    struct {
        uint8_t HDMI_TX_LINK_ACTIVE_STATUS:1;  // Bit 0
        uint8_t HDMI_TX_READY_STATUS:1;        // Bit 1
        uint8_t RESERVED:6;
    } bits;
    uint8_t raw;
};
```

### Register 3: DP_PCON_HDMI_POST_FRL_STATUS
- **DPCD Address**: `0x3036`
- **Purpose**: Reports the post-FRL-link-training status, including the trained bandwidth and whether training finished
- **Definition**:
```c
union hdmi_encoded_link_bw {
    struct {
        uint8_t FRL_MODE:1;                     // Bit 0
        uint8_t BW_9Gbps:1;                     // Bit 1
        uint8_t BW_18Gbps:1;                    // Bit 2
        uint8_t BW_24Gbps:1;                    // Bit 3
        uint8_t BW_32Gbps:1;                    // Bit 4
        uint8_t BW_40Gbps:1;                    // Bit 5
        uint8_t BW_48Gbps:1;                    // Bit 6
        uint8_t FRL_LINK_TRAINING_FINISHED:1;  // Bit 7
    } bits;
    uint8_t raw;
};
```

## 3. Exact Failure Condition ("has not finished")

**Source code** (from `link_dp_capability.c`, lines 1138-1148):

```c
// Intersect reported max link bw support with the supported link rate post FRL link training
if (core_link_read_dpcd(link, DP_PCON_HDMI_POST_FRL_STATUS,
        &hdmi_encoded_link_bw.raw, sizeof(hdmi_encoded_link_bw)) == DC_OK) {

    if (link->dpcd_caps.dongle_caps.dp_hdmi_regulated_autonomous_mode_support &&
        (!hdmi_tx_link_status.bits.HDMI_TX_READY_STATUS ||
         !hdmi_encoded_link_bw.bits.FRL_LINK_TRAINING_FINISHED)) {
        DC_LOG_WARNING("%s: PCON TX link training has not finished.\n", __func__);

        /* Link training not finished, ignore values from this DPCD reg. */
        return;
    }

    link->dpcd_caps.dongle_caps.dp_hdmi_frl_max_link_bw_in_kbps = intersect_frl_link_bw_support(
            link->dpcd_caps.dongle_caps.dp_hdmi_frl_max_link_bw_in_kbps,
            hdmi_encoded_link_bw);
    DC_LOG_DC("%s: pcon frl link bw = %u\n", __func__,
            link->dpcd_caps.dongle_caps.dp_hdmi_frl_max_link_bw_in_kbps);
}
```

**Failure condition breakdown**:
1. The PCON dongle reports `dp_hdmi_regulated_autonomous_mode_support = true` (read from DPCD `0x303C`)
2. **AND** either:
   - `HDMI_TX_READY_STATUS` (bit 1 of `0x303B`) is **0**
   - **OR** `FRL_LINK_TRAINING_FINISHED` (bit 7 of `0x3036`) is **0**
3. When both conditions are true, the driver logs the warning using `DC_LOG_WARNING` with `__func__`, producing:
   `[drm] read_and_intersect_post_frl_lt_status: PCON TX link training has not finished.`

## 4. Timeout / Retry Logic

**NO explicit timeout or retry exists inside `read_and_intersect_post_frl_lt_status()`.**

The function performs a **single-shot DPCD read** of the three registers and immediately evaluates the status. There is:
- No `for` loop for retries
- No `msleep()` or `udelay()` wait inside the function
- No counter tracking attempts
- No timeout value (e.g., 100ms, 500ms)

**Why the message appears 95+ times**: 
The function is called from `get_active_converter_info()` which is called from `retrieve_link_cap()` during **link detection**. Link detection is triggered repeatedly by:
- HPD (Hot Plug Detect) IRQ events
- Mode validation / display enable attempts
- Display state changes (e.g., resume from suspend, VT switch, compositor re-initialization)
- Polling loops in the amdgpu DM layer when a display fails to come up

Each trigger causes a new call to `retrieve_link_cap()` → `read_and_intersect_post_frl_lt_status()` → another warning log. There is **no backoff or rate-limiting** on the warning.

## 5. Fallback Behavior After Failure

When the failure condition is met, the function executes an **early return** at line 1147:

```c
/* Link training not finished, ignore values from this DPCD reg. */
return;
```

**Consequences**:
1. `link->dpcd_caps.dongle_caps.dp_hdmi_frl_max_link_bw_in_kbps` **is NOT updated** by `intersect_frl_link_bw_support()`.
2. The value retains whatever was previously set at line 1247-1249 in `get_active_converter_info()`:
   ```c
   link->dpcd_caps.dongle_caps.dp_hdmi_frl_max_link_bw_in_kbps =
       link_bw_kbps_from_raw_frl_link_rate_data(
           hdmi_color_caps.bits.MAX_ENCODED_LINK_BW_SUPPORT);
   ```
   This comes from the downstream port caps byte 2 (`MAX_ENCODED_LINK_BW_SUPPORT` field, bits 2:4 of DPCD offset 0x0082).

3. In `get_active_converter_info()` (lines 1253-1254):
   ```c
   if (link->dpcd_caps.dongle_caps.dp_hdmi_frl_max_link_bw_in_kbps > 0)
       link->dpcd_caps.dongle_caps.extendedCapValid = true;
   ```
   If the max FRL BW is 0, `extendedCapValid` remains false.

4. **No explicit fallback to TMDS is triggered by this function**. The mode validation in `dc_link.c` (`dp_active_dongle_validate_timing()`) has logic:
   ```c
   if (dongle_caps->dp_hdmi_frl_max_link_bw_in_kbps > 0) {
       // FRL converter path
   } else {
       // TMDS converter path: check pixel clock against dp_hdmi_max_pixel_clk_in_khz
   }
   ```
   So if `dp_hdmi_frl_max_link_bw_in_kbps` is 0 or the timing exceeds the FRL BW, the driver may fall back to TMDS validation. However, the PCON itself (in autonomous mode) is responsible for managing the HDMI link training with the sink; the driver only reads the status.

5. **The driver does NOT reduce lane count, link rate, or retry link training** for the PCON FRL path. The PCON is supposed to handle FRL link training autonomously. The driver merely checks if the PCON reports success.

## 6. `intersect_frl_link_bw_support()` Function

**Location**: `link_dp_capability.c`, lines 255-280
**Signature**: `static uint32_t intersect_frl_link_bw_support(const uint32_t max_supported_frl_bw_in_kbps, const union hdmi_encoded_link_bw hdmi_encoded_link_bw)`

**Behavior** (notable: it does NOT check `FRL_MODE` bit):
```c
/* Skip checking FRL_MODE bit, as certain PCON will clear
 * it despite supporting the link BW indicated in the other bits.
 */
if (hdmi_encoded_link_bw.bits.BW_48Gbps)
    supported_bw_in_kbps = 48000000;
else if (hdmi_encoded_link_bw.bits.BW_40Gbps)
    supported_bw_in_kbps = 40000000;
else if (hdmi_encoded_link_bw.bits.BW_32Gbps)
    supported_bw_in_kbps = 32000000;
else if (hdmi_encoded_link_bw.bits.BW_24Gbps)
    supported_bw_in_kbps = 24000000;
else if (hdmi_encoded_link_bw.bits.BW_18Gbps)
    supported_bw_in_kbps = 18000000;
else if (hdmi_encoded_link_bw.bits.BW_9Gbps)
    supported_bw_in_kbps = 9000000;
else if (hdmi_encoded_link_bw.bits.FRL_LINK_TRAINING_FINISHED)
    supported_bw_in_kbps = 0; /* This case should only get hit in regulated autonomous mode. */

return supported_bw_in_kbps;
```

This function is only called when `FRL_LINK_TRAINING_FINISHED` is already verified as set, so the `else if (FRL_LINK_TRAINING_FINISHED)` branch is effectively a safety net.

## 8. Root Cause Analysis for CH7218A

The CH7218A PCON adapter (used in UGREEN/DisplayPort-to-HDMI 2.1 dongles) appears to have a firmware or hardware behavior where:

1. It reports `REGULATED_AUTONOMOUS_MODE_SUPPORTED = 1` (DPCD 0x303C, bit 7)
2. But it either:
   - Never sets `HDMI_TX_READY_STATUS = 1` (DPCD 0x303B, bit 1), OR
   - Never sets `FRL_LINK_TRAINING_FINISHED = 1` (DPCD 0x3036, bit 7), OR
   - Both bits remain 0 indefinitely

This causes `read_and_intersect_post_frl_lt_status()` to **always** hit the failure path and emit the warning every time link capability is retrieved.

**Notable**: The AMD driver does not attempt to wait, retry, or force FRL link training. It only reads the status. The PCON is supposed to perform FRL link training autonomously when it detects an HDMI 2.1 sink. If the PCON never reports training finished, the driver has no recovery mechanism.

**Community reports**:
- Bazzite issue #4532
- Reddit r/linux_gaming (UGREEN Knight discussion)
- KDE bug 517051

AMD reportedly wants to test the CH7218A themselves (per Reddit comments), suggesting no upstream fix exists yet.
