# Research: CH7218 Dual-Display Stable vs Single-Display Unstable

## Executive Summary

The CH7218A PCON adapter exhibits stable 10-bit HDR 4K@120Hz behavior when both monitors are active (extend mode), but oscillates between 8-bit and 10-bit with an 8-bit black screen when it is the only active display. This research identifies multiple code paths in amdgpu DC that behave differently between single and dual display configurations, with the most critical being:

1. **DSC `is_frl` flag handling**: The `dc_dsc_config.is_frl` field is never set to `true` for PCON adapters, causing DSC to be unconditionally stripped from output timing during bandwidth validation for FRL PCONs. This means bandwidth validation is done against uncompressed bandwidth.
2. **DRAM clock (MCLK) policy**: By default, MCLK may not switch during vactive with a single display, but this behavior changes with dual displays. The `enable_dram_clock_change_one_display_vactive` debug option directly controls this.
3. **Link training retry and fallback logic**: The `perform_link_training_with_retries` function recalculates required bandwidth using `dc_bandwidth_in_kbps_from_timing()` after each fallback. If DSC configuration differs between validation attempts, bandwidth requirements change, potentially causing oscillation.
4. **Pipe split and power gating policy**: DC applies different pipe split (`MPC_SPLIT_AVOID_MULT_DISP`) and power gating (`disable_dsc_power_gate`) policies for single vs dual displays.

## 1. DP Link Training Behavior Differences

### 1.1 Link Training Fallback Logic

**File**: `drivers/gpu/drm/amd/display/dc/core/dc_link_dp.c` (older kernels) / `drivers/gpu/drm/amd/display/dc/link/protocols/link_dp_training_8b_10b.c` (newer kernels)

**Function**: `bool perform_link_training_with_retries(...)`

Key code path from commit `4de0bfe67bc98fb2a1115bc58ef3b9bc21717629` and later refinements:

```c
bool perform_link_training_with_retries(
    struct dc_state *state,
    struct pipe_ctx *pipe_ctx,
    const struct dc_link_settings *link_setting,
    int attempts, ...)
{
    struct dc_link_settings cur_link_settings = *link_setting;
    struct dc_link_settings max_link_settings = *link_setting;  // Added in 2022 fix
    ...
    for (j = 0; j < attempts; ++j) {
        status = dc_link_dp_perform_link_training(link, &pipe_ctx->link_res,
                                                  &cur_link_settings, skip_video_pattern);
        if (status == LINK_TRAINING_SUCCESS && !is_link_bw_low)
            return true;
        ...
        if (do_fallback) {
            uint32_t req_bw;
            uint32_t link_bw;

            decide_fallback_link_setting(link, &max_link_settings,
                                         &cur_link_settings, status);
            req_bw = dc_bandwidth_in_kbps_from_timing(&stream->timing);
            link_bw = dc_link_bandwidth_kbps(link, &cur_link_settings);
            if (req_bw > link_bw)
                break;  // FAIL: bandwidth too low after fallback
        }
    }
}
```

**Critical difference**: After each failed link training attempt, `dc_bandwidth_in_kbps_from_timing(&stream->timing)` is called to compare against the link bandwidth. If the timing's DSC flag is enabled or disabled inconsistently between attempts (due to validation vs actual configuration), the required bandwidth calculation changes, which could cause link training to alternate between success and failure.

### 1.2 Fallback Decision Function

**Function**: `static bool decide_fallback_link_setting(...)`

```c
static bool decide_fallback_link_setting(
    struct dc_link *link,
    struct dc_link_settings *max,
    struct dc_link_settings *cur,
    enum link_training_result training_result)
{
    switch (training_result) {
    case LINK_TRAINING_CR_FAIL_LANE0:
    case LINK_TRAINING_CR_FAIL_LANE1:
    case LINK_TRAINING_CR_FAIL_LANE23:
        if (!reached_minimum_link_rate(cur->link_rate)) {
            cur->link_rate = reduce_link_rate(cur->link_rate);
        } else if (!reached_minimum_lane_count(cur->lane_count)) {
            cur->link_rate = max->link_rate;
            cur->lane_count = reduce_lane_count(cur->lane_count);
        } else {
            return false;  // Cannot fall back further
        }
        break;
    case LINK_TRAINING_EQ_FAIL_EQ:
        if (!reached_minimum_lane_count(cur->lane_count)) {
            cur->lane_count = reduce_lane_count(cur->lane_count);
        } else if (!reached_minimum_link_rate(cur->link_rate)) {
            cur->link_rate = reduce_link_rate(cur->link_rate);
            max->link_rate = cur->link_rate;  // Prevent infinite loop
            cur->lane_count = max->lane_count;
        } else {
            return false;
        }
        break;
    }
}
```

**Relevant commit**: `Ilya.Bakoulin at amd.com`, May 2022 - "Fix possible infinite loop in DP LT fallback" (full hash: `997952851843935024962b51fc36c61b3c2d1ed4`)

### 1.3 Debug Options for Link Training

In `struct dc_debug_options` (from `drivers/gpu/drm/amd/display/dc/dc.h`):

## 2. DSC Configuration Differences

### 2.1 The `is_frl` Flag in DSC Config

**File**: `drivers/gpu/drm/amd/display/dc/dc_hw_types.h`

**Struct**: `struct dc_dsc_config`

```c
struct dc_dsc_config {
    uint32_t version_minor;
    bool ycbcr422_simple;
    int32_t rc_buffer_size;
#if defined(CONFIG_DRM_AMD_DC_DCN)
    bool is_frl; /* indicate if DSC is applied based on HDMI FRL sink's capability */
#endif
    bool is_dp;  /* indicate if DSC is applied based on DP's capability */
    ...
};
```

**Patch history**: Added in November 2021 by Fangzhi Zuo (`Jerry.Zuo at amd.com`) in the patch series "Add DP-HDMI FRL PCON Support in DC".

### 2.2 DSC Stripping in PCON Validation

**File**: `drivers/gpu/drm/amd/display/dc/core/dc_link.c` (older) / `drivers/gpu/drm/amd/display/dc/link/link_validation.c` (newer)

**Function**: `static bool dp_active_dongle_validate_timing(...)`

```c
static bool dp_active_dongle_validate_timing(
    const struct dc_dongle_caps *dongle_caps,
    const struct dc_crtc_timing *timing)
{
    ...
#if defined(CONFIG_DRM_AMD_DC_DCN)
    if (dongle_caps->dp_hdmi_frl_max_link_bw_in_kbps > 0) { // DP to HDMI FRL converter
        struct dc_crtc_timing outputTiming = *timing;

        if (timing->flags.DSC && !timing->dsc_cfg.is_frl)
            /* DP input has DSC, HDMI FRL output doesn't have DSC,
             * remove DSC from output timing */
            outputTiming.flags.DSC = 0;
        if (dc_bandwidth_in_kbps_from_timing(&outputTiming) >
            dongle_caps->dp_hdmi_frl_max_link_bw_in_kbps)
            return false;
    }
#endif
    ...
}
```

**CRITICAL FINDING**: The check `if (timing->flags.DSC && !timing->dsc_cfg.is_frl)` strips DSC from the output timing for bandwidth validation. However, **NEVER FOUND** any code path where `is_frl` is explicitly set to `true` for PCON adapters in the amdgpu DC codebase. This means:

- For any PCON adapter, DSC is always stripped during `dp_active_dongle_validate_timing`
- Bandwidth validation is performed against uncompressed timing
- If the uncompressed 4K@120Hz 10-bit timing exceeds the PCON's FRL bandwidth cap, the mode would be rejected

**The oscillation hypothesis**: If the actual stream commit path applies DSC (because the GPU DSC encoder is enabled), but validation strips it, the link training sees two different bandwidth requirements:
1. Validation path: no DSC (high bandwidth) → may reject or force fallback
2. Commit path: DSC enabled (lower bandwidth) → may succeed
This mismatch could cause oscillation between 8-bit and 10-bit as the driver retries different configurations.

### 2.3 DSC Bandwidth Calculation

**File**: `drivers/gpu/drm/amd/display/dc/dsc/dc_dsc.c`

**Function**: `uint32_t dc_bandwidth_in_kbps_from_timing(const struct dc_crtc_timing *timing)`

```c
uint32_t dc_bandwidth_in_kbps_from_timing(const struct dc_crtc_timing *timing)
{
    if (timing->flags.DSC) {
        return dc_dsc_bandwidth_in_kbps_from_timing(&timing->dsc_cfg, timing);
    } else {
        return dc_bandwidth_in_kbps_from_timing_no_dsc(timing);
    }
}
```

When DSC is stripped (`outputTiming.flags.DSC = 0`), this function falls through to the uncompressed path, multiplying pixel clock by bits-per-pixel. For 4K@120Hz 10-bit HDR, this could easily exceed 48Gbps FRL6 bandwidth.

### 2.4 DSC Power Gating

From `struct dc_debug_options`:
- `bool disable_dsc_power_gate;` — If false (default), DSC power gating is enabled. With a single display, power gating may aggressively turn off DSC hardware when not needed, causing re-initialization delays.

## 3. Pixel Clock / Display Clock / DML Differences

### 3.1 Single-Display DRAM Clock Policy

**File**: `drivers/gpu/drm/amd/display/dc/dcn20/dcn20_resource.c` (and similar for DCN21/30/31)

In `dcn20_validate_bandwidth_fp()`:

```c
context->bw_ctx.dml.soc.allow_dram_clock_one_display_vactive =
    dc->debug.enable_dram_clock_change_one_display_vactive;
```

**Default behavior**: By default, `enable_dram_clock_change_one_display_vactive` is `false`, meaning **MCLK is NOT allowed to change during vactive when only one display is connected**. With dual displays, this restriction is lifted.

**Impact**: With dual displays, MCLK stays at a higher level (~1000MHz) because it must service both displays. With a single display, MCLK drops to lower levels (~96MHz or ~456MHz) because:
1. The SMU firmware sees lower memory bandwidth demand
2. DCN explicitly allows lower MCLK states with single display
3. Lower MCLK may cause timing issues for high-bandwidth DP link training

### 3.2 DML (Display Mode Library) Clock Validation

**File**: `drivers/gpu/drm/amd/display/dc/dml/`

The DML calculates required DISPCLK, DPPCLK, and DRAM bandwidth. Key structures:
- `display_e2e_pipe_params` — per-stream pipe parameters
- `display_mode_lib` — DML state

**Difference**: In dual-display mode, DML must account for total bandwidth across all active pipes. In single-display mode, pipe split policy may differ:

```c
enum pipe_split_policy {
    MPC_SPLIT_DYNAMIC,              // Default
    MPC_SPLIT_AVOID,                // Never split
    MPC_SPLIT_AVOID_MULT_DISP,      // Avoid split only with multiple displays
};
```

With `MPC_SPLIT_DYNAMIC` (default), DC may split a single 4K@120Hz stream across two pipes when bandwidth is tight. This changes the DPPCLK requirements and DSC slice allocation.

### 3.3 Relevant Debug Options

From `struct dc_debug_options`:
- `bool enable_dram_clock_change_one_display_vactive;` — **KEY**: Set to `true` to allow MCLK changes with single display
- `unsigned int min_disp_clk_khz;` — Override minimum DISPCLK
- `unsigned int min_dpp_clk_khz;` — Override minimum DPPCLK
- `unsigned int min_dram_clk_khz;` — Override minimum DRAM/MCLK
- `bool disable_min_fclk;` — Disable minimum FCLK enforcement
- `bool force_single_disp_pipe_split;` — Force pipe split even with single display

## 4. Power State Differences

### 4.1 MCLK Drop with Single Display

**Observation**: MCLK drops from ~1000MHz (dual) to ~96MHz or ~456MHz (single).

**Explanation**: The amdgpu DPM (Dynamic Power Management) firmware lowers MCLK when it detects lower memory bandwidth demand. With a single 4K@120Hz display, even though bandwidth is high, the firmware may mispredict or the DC watermark settings may report lower urgency.

**File**: `drivers/gpu/drm/amd/display/dc/clk_mgr/` (various ASIC-specific clock managers)

The `update_clocks()` callback in each clk_mgr calls into SMU firmware to request MCLK levels. If the watermark calculations indicate that a lower MCLK is sufficient, the SMU drops to a lower state.

### 4.2 Power Gating and State Transitions

From `struct dc_debug_options`:
- `bool disable_dpp_power_gate;` — DPP (Display Pipe and Plane) power gating
- `bool disable_hubp_power_gate;` — HUBP (Hubp) power gating
- `bool disable_dsc_power_gate;` — DSC encoder power gating
- `bool disable_mem_low_power;` — Memory low-power modes
- `bool disable_idle_power_optimizations;` — Disable all idle power optimizations

**Hypothesis**: When the second monitor is disconnected, DC performs a state transition that:
1. Re-allocates pipes
2. Re-computes DSC configuration
3. Re-trains the DP link
4. Requests lower MCLK from SMU

If any of these steps races with another (e.g., DSC power gate turning off while link training is in progress), the link training may see transient failures.

### 4.3 SubVP and FAMS

From `struct dc_debug_options`:
- `bool force_disable_subvp;` — Disable Sub-Viewport (SubVP) feature
- `bool force_subvp_mclk_switch;` — Force SubVP MCLK switching
- `bool disable_fams;` — Disable Firmware Assisted MCLK Switching (FAMS)

SubVP and FAMS are DCN3.x+ features that enable MCLK switching during vblank. If these are enabled, single-display MCLK transitions may interfere with DP link stability.

## 7. Root Cause Hypotheses

### Hypothesis A: DSC Configuration Mismatch + Link Training Oscillation

**Mechanism**:
1. When CH7218 is the only active display, amdgpu DC attempts to enable DSC for 4K@120Hz 10-bit
2. `dp_active_dongle_validate_timing()` strips DSC because `dsc_cfg.is_frl == false`
3. Uncompressed bandwidth exceeds CH7218 FRL cap, so validation fails or requests higher link rate
4. Link training succeeds at higher rate but with DSC enabled (actual commit path uses DSC)
5. A subsequent mode validation sees the mismatch and tries 8-bit fallback
6. 8-bit mode has lower bandwidth, passes validation without DSC, but results in black screen
7. Driver detects failure and retries 10-bit, causing oscillation

**Why dual-display is stable**:
- With dual displays, the DML validation and pipe assignment are computed together
- The total bandwidth constraints force DSC to be applied consistently to both displays
- The `is_frl` stripping behavior is bypassed or compensated because dual-display validation follows a different path (MST fairness algorithm)

### Hypothesis B: MCLK Power State Interference

**Mechanism**:
1. Single display allows MCLK to drop to ~96MHz or ~456MHz
2. Lower MCLK increases memory latency for display buffer fetches
3. DP link training requires consistent AUX channel and DPCD register access timing
4. Memory latency jitter causes intermittent DPCD read failures during link training
5. Driver falls back to 8-bit as a "safe" mode
6. With dual displays, MCLK stays at ~1000MHz, avoiding the timing issue

### Hypothesis C: Pipe Reassignment Race

**Mechanism**:
1. Disconnecting second monitor triggers DC pipe reassignment
2. DSC encoder is power-gated during the transition
3. New DSC configuration is computed for the single display
4. Race condition: link training starts before DSC encoder is fully powered and programmed
5. PCON receives uncompressed stream, shows black screen
6. Driver retries with 8-bit (smaller bandwidth, no DSC needed)

## 10. NOT FOUND Items

1. **Exact assignment of `dsc_cfg.is_frl = true`**: Searched extensively across `drivers/gpu/drm/amd/display/` in git history. The field is checked (`!timing->dsc_cfg.is_frl`) in `dp_active_dongle_validate_timing()`, but no code path was found that sets `is_frl = true` for HDMI FRL sinks or PCON adapters. It appears to always be `false` (zero-initialized), meaning DSC is always stripped for FRL PCON validation.

2. **Direct `amdgpu_dc_debug_mask` bit for `enable_dram_clock_change_one_display_vactive`**: This field exists in `struct dc_debug_options` but is NOT mapped to a module parameter bit in `enum amdgpu_dc_debug_flags`. It can only be set via kernel patch or DC internal defaults.

3. **Specific `disable_dsc_power_gate` module parameter bit**: Not found in `amd_shared.h` debug flags enum. May only be settable via `struct dc_debug_options` defaults.
